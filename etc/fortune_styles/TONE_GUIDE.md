# 운세 톤 가이드 (Yegamssi Fortune Styles)

7개 톤별 작성 규칙과 샘플. 기본/유머 톤은 확정, 나머지 5개는 본 문서 샘플 기반 사용자 검토 후 확정.

---

## 공통 구조
- 동일 데이터 구조: code/type/text/weight
- 동일 코드 매트릭스: 6 카테고리 × 6 티어 × 오행/날씨
- text만 톤에 맞게 변환

### type 4종 역할 기준
`fortune_ko_base.csv` 기준으로 `type`은 하나의 운세 문장을 구성하는 4개 조각이다. 각 톤을 만들 때도 이 역할을 유지해야 한다.

### ELEM 텍스트 고유성 기준 (2026-05-17 확정)

BASE 코드(`decision_A` 등)와 ELEM 코드(`decision_A_geum_df_earth` 등)의 텍스트 작성 방식은 다르다.

#### BASE 코드 텍스트
- 코드 형식: `{category}_{tier}` (예: `decision_A`)
- intro/effect/action: 오행·날씨와 무관하게 **카테고리×티어** 단위로 1개 작성
- state: `STATE_ELEM[tier][ohaeng_dfex]` + `" "` + `STATE_WEATHER[tier][weather]` 2절 조합

#### ELEM 코드 텍스트
- 코드 형식: `{category}_{tier}_{ohaeng}_{dfex}_{weather}` (예: `decision_A_geum_df_earth`)
- intro/effect/action: **`(category × tier × ohaeng)` 조합마다 독립 텍스트** 작성
  - 6 카테고리 × 6 티어 × 5 오행 = **540개** 고유 텍스트 (type 당)
  - dfex(df/ex)와 weather는 intro/effect/action을 구분하지 않는다 → 같은 ohaeng 내 dfex·weather 변형 행은 동일 텍스트 사용
  - BASE 텍스트에 suffix를 이어붙이는 방식 **금지**
- state: BASE와 동일하게 `STATE_ELEM` + `STATE_WEATHER` 2절 조합 (카테고리 독립)

#### 오행별 텍스트 방향성
| 오행 | 키워드 | 문장 톤 방향 |
|---|---|---|
| geum (금) | 결단·날카로움·집중 | 명확하고 직접적인 표현 |
| hwa (화) | 직관·열정·감각 | 빠르고 감각적인 표현 |
| mok (목) | 성장·유연·확장 | 유연하고 가능성 중심 표현 |
| su (수) | 흐름·타이밍·적응 | 흐름과 타이밍 중심 표현 |
| to (토) | 안정·신중·균형 | 차분하고 안정 중심 표현 |

#### 티어별 텍스트 방향성
| 티어 | 점수 | 방향 |
|---|---|---|
| A | 80+ | 최상 — 과감한 행동 권장 |
| B | 63~79 | 상 — 안정적 진행 권장 |
| B1 | 50~62 | 중상 — 신중한 점검 권장 |
| C | 38~49 | 중하 — 보수적 대응 권장 |
| C1 | 25~37 | 하 — 핵심만 최소 행동 |
| D | 0~24 | 최하 — 행동 보류·휴식 권장 |

### 조합 규칙
- `intro`는 독립적으로 읽혀도 자연스러워야 한다.
- `state`는 세부 변수 설명용이다. 단순 감성 문구나 캐릭터 말투만 넣으면 최종 운세가 흐려진다.
- `effect`는 상태 설명을 반복하지 말고 결과/영향을 말해야 한다.
- `action`은 반드시 행동으로 이어져야 한다. `좋습니다`, `괜찮습니다` 같은 평가만 쓰면 안 된다.
- 4개 조각을 합쳤을 때 `판단 → 상태 → 영향 → 행동` 순서로 읽혀야 한다.
- 같은 기본 문장에 톤 꼬리만 붙이는 방식은 금지한다.
- 톤은 달라도 `type`의 기능은 바뀌면 안 된다.

### 파일 용량 기준
- 각 톤 CSV 파일은 **900KB 이하**로 유지한다.
- 기준은 Windows 파일 속성의 실제 파일 크기이며, 스크립트 검증 시 `900,000 bytes` 이하로 판단한다.
- 용량 초과 시 문장 품질을 해치지 않는 범위에서 `state` 문장과 반복 설명을 먼저 줄인다.
- 용량을 줄이기 위해 `code/type/weight` 구조를 변경하거나 필요한 행을 삭제하지 않는다.
- 업로드 전에는 행 수, 기준 코드 매트릭스, 중복 여부와 함께 파일 크기를 반드시 확인한다.

### type별 금지 패턴
- `intro`: 행동 지시만 쓰는 것 금지. 예: `오늘은 쉬세요`
- `state`: 결과나 행동 조언으로 끝내는 것 금지. 예: `그러니 움직이세요`
- `effect`: 상태 묘사만 반복하는 것 금지. 예: `기운이 좋고 흐름이 좋습니다`
- `action`: 추상적 감상만 쓰는 것 금지. 예: `마음이 따뜻해질 수 있습니다`

## 1. 기본 (✅ 확정)
원본 deduped 파일. 정보 100%, 무난한 어조.
- `오늘 판단력이 최고조로 올라 복잡한 상황도 명료하게 보이는 날입니다`
- `오늘 결정 기운이 완전히 막혀 있어 모든 선택을 내일 이후로 미루는 것이 맞습니다`

## 2. 유머 (✅ 확정)
짤방/밈 인용, 중간 강도, 슬랭 자유. 금지어: `각`, `운빨`, `★`.
- 일상 비유 (지갑/이불/카톡/통장), 게임/IT 비유, 자조/시나리오
- 어휘 풀: `풀가동`, `풀충전`, `만렙`, `최상급`, `ㄹㅇ`, `갓생/갓컨`, `오운완`, `현타`, `스불재`

## 3. 츤데레 (⏸ 검토 대기)
**규칙:** 표면=무뚝뚝/부정문, 속내=챙김. 직접 표현 회피.
**어미:** `~인 거 아니야`, `~라고`, `~한 건 아니지만`, `~인 거지 뭐`, `착각하지 마`
**금지:** 너무 다정한 표현, 직접적 응원

## 4. 시니컬 (⏸ 검토 대기)
**규칙:** 건조한 사실 진술, 비꼼, 기대 낮춤. 과장된 좋음/나쁨도 시큰둥하게.
**어미:** `~이긴 하다`, `~일 뿐`, `~할 텐데 뭐`, `~다고 한들`, `어차피 ~`
**금지:** 격려성 표현, 느낌표

## 5. 감성 (⏸ 검토 대기)
**규칙:** 시적 표현, 자연 메타포(별·바람·햇살·꽃), 따뜻한 온도. 위로/공감 톤.
**어미:** `~입니다`, `~네요`, `~지요`, `~겠어요`
**금지:** 슬랭, 약자, 과한 감탄

## 6. 사극 (⏸ 검토 대기)
**규칙:** 고어체 어미 + 자연스러운 현대어 혼용. 한자 괄호 금지. 운명/기운/천기 어휘 활용.
**어미:** `~하시오`, `~하느니라`, `~하오`, `~이리라`, `~마땅하오`, `~할지어다`
**금지:** 현대 슬랭, 영어, 숫자/기호 표기, 한자 괄호(한자어·단어는 허용)

## 7. AI (⏸ 검토 대기)
**규칙:** 대화체 + AI 분석 말투. "분석 결과", "데이터 기준" 등 AI 어휘 포함. 수치/조건은 자연스럽게 녹임.
**어미:** `~에요`, `~입니다`, `~합니다` (대화체 유지)
**금지:** 슬랭, 비유, 리포트 형식 대괄호 라벨, 느낌표
